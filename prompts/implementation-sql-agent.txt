## Regras de Implementação

### Migrations
- SEMPRE incluir scripts de UP e DOWN (rollback)
- Versionamento sequencial (V001_, V002_)
- Transações explícitas (BEGIN TRAN / COMMIT)
- IF NOT EXISTS para idempotência em DDL
- Comentários com propósito em cada migration

### Tabelas
- Naming: snake_case, plural (users, orders, order_items)
- Toda tabela com: id (PK auto-increment ou UUID), created_at, updated_at
- Soft delete via deleted_at (nullable DATETIME) quando aplicável
- Foreign keys com ON DELETE adequado (CASCADE, SET NULL, RESTRICT)
- CHECK constraints para validação de domínio
- Indexes em FKs e colunas frequentemente filtradas

### Queries & Procedures
- SEMPRE usar parâmetros (@param) — NUNCA concatenar strings
- CTEs para queries complexas (melhor que subqueries aninhadas)
- Stored procedures com: tratamento de erro (TRY/CATCH), transaction management, output params para status
- Views para queries complexas recorrentes
- Paginação eficiente (OFFSET/FETCH ou keyset pagination)

### Performance
- Indexes compostos seguindo order de seletividade
- INCLUDE columns em indexes para covering queries
- Statistics atualizados para query optimizer
- EXPLAIN/execution plan verificado para queries críticas
- Evitar SELECT * — listar colunas explicitamente
- Batch operations para INSERT/UPDATE massivo

### Segurança
- Permissions granulares (GRANT SELECT, INSERT, EXECUTE)
- Roles aplicados (app_read, app_write, app_admin)
- Encrypt columns sensíveis (senhas, PII)
- Audit table para operações críticas (INSERT trigger)
- Nunca expor connection strings em código
